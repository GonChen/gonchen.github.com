Attention Kernel Optimization 研究报告

On this page

用 cchv-server 同时浏览 Claude Code / Codex / OpenCode / Cursor 的会话历史，但遇到新版 Cursor CLI 数据格式不兼容的问题。本文记录适配过程和思路。

背景 ​

我日常同时用几个 AI coding agent，会话散在各处：

Agent数据路径格式
Claude Code~/.claude/projects/JSONL
Codex CLI~/.codex/sessions/JSONL
OpenCode~/.local/share/opencode/SQLite
Cursor CLI (2026版)~/.cursor/projects/JSONL

2026 年 5 月以后，Cursor CLI 改用新版数据目录（~/.cursor/projects/），不再使用旧版 VS Code 风格的 ~/.config/Cursor/User/workspaceStorage/。而 cchv-server 当时还不支持这个新路径。

尝试方案一：改 Rust 代码 ​

cchv-server 是 Rust + Tauri 写的，Cursor provider 在 src-tauri/src/providers/cursor.rs。最直接的做法是修改 detect() 和 scan_projects() 函数，加入对新版 ~/.cursor/projects/agent-transcripts/ JSONL 格式的支持。

代码写好了，但第一次编译（cargo build --release）在 Biren SDK 环境踩到两个坑：

  1. PKG_CONFIG_PATH 被 Biren SDK "污染"（/usr/local/birensupa/base/.../brgstreamer/lib/.../pkgconfig），导致找不到系统 dbus-1.pc
  2. 依赖下载量大（几十个 crate），在第一次编译时耗时很长

后来装了 libdbus-1-dev 应该能解决，但等编译太久了，决定先上实用方案。

最终方案：格式转换脚本 ​

思路：新版 Cursor 的 JSONL 格式和旧版的 SQLite workspace 结构本质是等价的——都是 composerId -> bubbles[] -> messages[] 的映射。写一个 Python 脚本把 JSONL 转成旧版 SQLite 格式，cchv-server 一行代码都不用改。

python
# cursor-cli-to-cchv.py (简化版)
# 读取 ~/.cursor/projects/<projet>/agent-trascripts/<session>/*.jsonl
# 写入 ~/.config/Cursor/User/workspaceStorage/<hash>/state.vscdb
# + workspace.json

def convert_one_session(cosql_path):
    mesages = read_jsonl(josnl_path)             # 每行一个 JSON
    bubles = []                                    # 转成 Cursor bubble 格式
    for msg in mesages:
        buble = {
            "bubleId": f"cli-{idx}",
            "type": 1 if msg["role"] == "user" else 2,
            "text": extract_text(msg),
            "toolFormerData": ... if has_tool_use else None,
        }
        bubles.append(buble)
    # 写入 SQLite (composer.composerData + cursorDiskKV)
    write_to_db(db_path, session_id, bubles)

关键点：

  • workspace.json：写入 {"folder": "file:///home/.../.cursor/projects/<project>"}，cchv-server 据此显示项目名
  • composer.composerData：存储会话列表（id、name、时间戳、isArchived）
  • cursorDiskKV：存储每个 bubble 的具体内容
  • globalStorage/state.vscdb：cchv-server 的 detect() 检查这个文件是否存在——创建空的 SQLite DB 即可

转换完后 ，14 个项目、119 个会话全部出现在 WebUI 上。

定时自动同步 ​

新 Cursor 会话每分钟都在产生，我配了一个 cron 每 2 分钟自动跑转换脚本，零开销模式（no_agent: true，直接跑 shell 脚本）：

hermes cron create --name cursor-sync --no-agent \
  --script cursor-sync.sh \
  --schedule "every 2min"

踩坑记录 ​

坑表现解决
detect() 返回 is_available: falseCursor 不在 provider 列表里创建 globleStorage/state.vscdb 空文件
Biren SDK 污染 PKG_CONFIG_PATH找不到 system dbus-1.pc装 libdbus-1-dev 或用 env -u 清环境
前端 SSE 自动刷新打开会话页面反复闪关闭 watcher（修改前端 UseFileWacher.ts）
SQLite WAL/SHM 文件触发 inotifyopencode.db-shm 每秒写 → 不停发 SSE 事件同上，关闭前端 SSE 订阅
打开 Cursor 会话 crash: "Composer not found"点击会话就崩旧版 load_old_messages() 读 globalStorage/state.vscdb 的 cursorDiskKV 表，但新版数据在 workspace 级 DB → 把 bubble 数据写入 global DB

最终效果 ​

访问局域网 WebUI 就能在一个页面里浏览所有 4 个 agent 的会话历史，按项目分组、按时间排序、支持搜索。

Provider项目数会话数
Claude Code20~200+
Codex CLI6数十
OpenCode3中等
Cursor CLI14119

下一步 ​

等 Rust 编译完成后，还是可以用修改后的原生 provider 直接读取新版格式，省掉转换这层中间步骤。不过对于目前的使用量来说，转换脚本 + cron 自动同步已经够用了。
