原文：Dynamic Workflows in Claude Code: Complete Guide
作者：Codez (@0xCodez) | 发布时间：2026-06-03 | 点赞：1,015 | 转发：140

核心问题 ​

"大多数 Claude Code 用户还在手写工作流……10 个开发者中有 9 个从未尝试过 Dynamic Workflows，尽管它两周前就已经发布了。"

Dynamic Workflows 于 2026 年 5 月 28 日 随 Claude Code 发布。默认的 harness 能应付大多数编码任务，但在长运行、大规模并行、高结构化或对抗性工作中就会力不从心。Dynamic Workflows 让 Claude 为你的具体任务编写自己的 JavaScript 协调代码——用 1 个 workflow 取代 50 条 prompt。

Part 1 · 心智模型 ​

1. 什么是 Dynamic Workflow？ ​

Dynamic Workflow 是 Claude 为任务编写的一个自定义协调程序——一个 JavaScript 文件，包含特殊的函数来生成和协调子代理（subagent），加上标准 JavaScript（Math、JSON、Array）在代理之间处理数据。

默认 harness 无法做到的 3 件事：

能力说明
每个代理独立上下文每个子代理拥有自己的上下文窗口和一个聚焦的目标，不会有交叉污染
每个代理可选模型workflow 可以为每个子代理选择不同的模型——Opus 做复杂推理、Haiku 做廉价探索、Sonnet 做中间层
每个代理独立等级Worktree（隔离的 git checkout）或 remote（无 checkout）

如何启动： 直接告诉 Claude（"make a workflow that..."）或使用触发词 ultracode。如果中断，恢复时可以从中断处继续。

2. Workflows 解决的 3 种失败模式 ​

失败模式描述
Agentic laziness（代理懒惰）Claude 在完成复杂任务前提前停止。例如安全审查只检查了 20/50 项，剩下报"已处理"
Self-preferential bias（自我偏好偏见）让 Claude 验证或评判自己的输出时，它会偏好自己的结果
Goal drift（目标漂移）经过多轮对话后原始目标逐渐丢失，第 47 轮时"不要做 X"的约束已经消失了

"Workflow 通过结构化的方式解决这三个问题：独立的 Claude 实例，各自拥有自己的上下文、聚焦的目标和隔离的状态。"

3. 静态 vs 动态 Workflows ​

静态 Workflows动态 Workflows
通用的——写一次要应付所有边界情况为具体的任务量身定制
必须保守适应你的上下文（代码、文档、数据）
无法知道你的代码存在可以读取你的计费代码，对照供应商文档检查特性，按你的用量定价

示例： 两者都能搜索，但动态 workflow 会围绕你的上下文自我塑造——在得出答案后再运行一个对抗性检查"为什么不应该迁移"。

4. 核心 API ​

三个函数完成大部分工作：

  • parallel() —— 屏障（barrier）：分发后在返回前等待所有结果。需要所有结果才能继续时用这个。
  • pipeline() —— 流式（streaming）：每个项目独立流过每个阶段。总体更便宜更快。项目之间不依赖时用这个。
  • agent() —— 生成一个拥有自己上下文的子代理。

"选择规则：我需要在做下一步之前拿到所有结果吗？是 → parallel()。否 → pipeline()（更便宜、更快）。"

Part 2 · 6 种模式 ​

5. Classify-and-Act（分类并行动） ​

一个分类器代理决定任务类型，然后将任务路由到不同的代理或行为。

适用场景：

  • 任务是异构的——不同子类型需要不同处理方式
  • 你想只在复杂度高的地方使用昂贵的模型
  • 任务的分解不是 trivial 的，让模型决定分解方式有好处

示例： "解释 auth 模块的工作原理。"分类器先读取代码库，估算复杂度，然后将 10 个文件的小模块路由给 Sonnet，100 个文件的大模块路由给 Opus。

6. Fan-Out-and-Synthesize（分发并合成） ​

把任务拆成许多小步骤 → 在每个步骤上并行运行一个代理 → 将结果合成为一个答案。

为什么它占主导： 解决了"一次性做太多事"的问题。每个子代理只看自己的部分，编排器永远不会被 50 个无关的细节分散注意力。

适用场景：

  • 你有一个可以清晰枚举的工作项列表（50 个文件、200 个端点、100 条审查）
  • 每个项目是独立的
  • 你想要一个合并的答案，而不是一堆零散报告

7. Adversarial Verification（对抗性验证） ​

为每个生成的代理，再运行一个独立的代理来对抗性地验证其输出。验证者从未看过原始工作。

最重要用途：

  • 事实核查 —— 每个事实陈述都获得自己的验证子代理
  • 代码审查 —— 作者代理写修复，评审代理（独立上下文）审查
  • 质量门禁 —— 在任何产物发布前，对抗者找其最薄弱的环节

"配对规则：验证者应该只知道评分标准和产物，不知道是谁生产的。"

8. Generate-and-Filter（生成并过滤） ​

生成大量想法 → 按评分标准/验证过滤 → 去重 → 只返回最高质量的。

优势场景：

  • 头脑风暴 —— 30 个产品名称 → 验证者剔除陈词滥调和商标冲突 → 你得到 3 个
  • 假设生成 —— 5 种方案，每种按约束条件打分
  • 方案设计

关键洞察： 问 Claude"最好的答案"会让 Claude 过早承诺。Generate-and-filter 让 Claude 在每一个选项都经历过挑战后才做出承诺。

9. Tournament（锦标赛） ​

生成 N 个代理用不同方法尝试同一任务 → 两两比较结果，直到选出胜者。

为什么它比打分排序好： 在一个 prompt 中对 1000 个项目排序会降低质量且超出上下文限制。锦标赛把淘汰赛分到一个个全新的代理中，每个代理只比较两个项目。

适用场景： 品味型排序、设计选择、候选人筛选、内容优先级排序。

10. Loop Until Done（循环直到完成） ​

适用于工作量未知的任务：循环生成代理，直到满足停止条件。每次迭代检查进度，必要时调整方向。

适用场景：

  • 重构一个你不知道有多少个调用点的函数
  • 渐进式性能优化："每次迭代让这个查询快 5%，直到无法再优化"
  • 批量处理 AI 生成的翻译，每批 100 条，处理完继续下一批

关键实现细节： 循环代理接收剩余任务列表 + 截止状态，只看到未完成的工作。

Part 3 · 实战案例 ​

"静态 workflow 就像模板，动态 workflow 就像建筑师亲自为你设计房子，看完你的土地、日程和风格偏好之后。你得到的是只能为你工作一次的东西。"

AI 审计分析 ​

任务： 审查 50 个 AI 工具的计费——找出隐藏费用、可疑的自动续费条款、以及"取消门口"的障碍。

结果： 一个单一的工作流定义生成了 21 个独立的子代理会话，分析了 10,000 页的文档、TOS 及定价页面，最终给出了一个包含价格对比和节省金额的报告。

"手动完成这个工作量是 50 个 prompt，每个 prompt 都需要你复制粘贴、修复问题、手动切换模型。一个 workflow 完成了全部。"

目标：连续 10 次正确的端到端测试 ​

方法： 让一个检查举证的代理失败一次 → 测试框架的代理修复一次 → 重试循环。

结果： 在第 11 次尝试时，测试通过了。整个过程中自动化代理自己调试、修复和重试，无需人工干预。

14 步快速上手 ​

  1. 开始之前： 确定你的任务是否真的需要 workflow——先尝试一个常规 prompt
  2. 选择一个模式： 回顾 6 种模式，找到最接近的
  3. 定义每个代理角色： 每个代理应该有一个动词（分析、比较、验证、生成）
  4. 命名输入/输出变量： 每个 stage 的输入和输出要清晰地命名
  5. 设置评分标准： 验证器的标准必须在代理看到产物之前定义
  6. 选择隔离等级： 每个代理需要 worktree（需要 git 隔离）还是 remote（不需要）
  7. 精心设计系统 prompt： 对每个子代理，告诉它要忽略什么和关注什么同样重要
  8. 先写核心骨架： 不用着急写 14 步——先做 2 个代理的工作流跑通
  9. 添加错误处理： agent 可能失败。你的 overlap 处理了吗？
  10. 异步 vs 同步： 所有任务可以并行吗？还是有些任务有依赖？
  11. 边界测试： 如果输入是空的、巨大的、或恶意格式的，workflow 会怎样？
  12. 观察指标： 你的监控指标是什么？信息素（subagent 产出的模式）、退出代码、还是时间？
  13. 迭代： 一个 workflow 很少在第一次就完美。Discord 社区有 pattern gallery
  14. 注意： 你仍然在一个 context window 中运行——不要试图一次自动化整个代码库
