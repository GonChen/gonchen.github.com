X 社区热议：Attention Kernel 优化趋势 (2026-05)

Attention Kernel Optimization 研究报告

On this page

配置看起来完全正确，阈值也已超限，但自动压缩就是不动。查到最后，原来是一个很少被注意的边界条件。

自动压缩是 Pi Agent 最省心的特性之一，但当你真正需要它的时候，它可能因为一个意想不到的原因静默失效。这篇文章记录了一次从配置检查、源码追踪到最终定位的完整排查过程，希望能帮到遇到类似问题的朋友。

背景 ​

我一直在用 Pi 作为日常的 AI 编程助手。作为一个终端内的 coding agent，Pi 的上下文管理一直是让我最省心的特性——它内置了自动压缩（auto-compaction）机制，能在会话增长到接近模型上下文窗口时，自动对早期对话做结构化摘要，保留近期内容完整。

但最近在跑一个超长会话时，我发现上下文已经撑到了 418K tokens，远超模型 deepseek-v4-flash 的 300K 上下文窗口，压缩却一次都没有触发。

排查过程 ​

第一步：验证配置 ​

检查 ~/.pi/agent/models.json，模型的 contextWindow 配置是明确的：

json
{
  "id": "deepseek-v4-flash",
  "contextWindow": 300000,
  "maxTokens": 64000
}

Pi 的 model-registry.js 在加载自定义 provider 的模型时确实会使用这个值：

javascript
contextWindow: modelDef.contextWindow ?? 128000,

配置本身没问题。

第二步：检查会话文件 ​

读取会话 JSONL 文件，确认实际的 token 用量确实远超阈值：

最后一条 assistant 消息的 usage:
totalTokens = 418554

自动压缩的触发逻辑在 compaction.js 中的 shouldCompact()：

javascript
export function shouldCompact(contextTokens, contextWindow, settings) {
    if (!settings.enabled) return false;
    return contextTokens > contextWindow - settings.reserveTokens;
}

代入数值：418554 > 300000 - 16384 = 283616 → true

按说应该触发，但会话文件中没有任何 compaction 类型的条目。

第三步：追踪触发时机 ​

我深入 Pi 的源码，追踪 _checkCompaction() 的调用路径。关键发现是——压缩检查只在每一"轮"（turn）结束时执行一次，而不是在每轮内的每个 tool call 之后。

代码路径：

_runAgentPrompt() → agent.prompt(messages) → agent loop 结束
  → _handlePostAgentRun()
    → _checkCompaction(_lastAssistantMessage)

_lastAssistantMessage 在 _handleAgentEvent() 中每次有新 assistant 消息时都会更新，记下最后一条：

javascript
if (event.message.role === "assistant") {
    this._lastAssistantMessage = event.message;
}

那问题来了：如果这最后一轮中，最后一条 assistant 消息的 token count 是正常的（超过 283K），压缩就应该触发。

第四步：发现中断场景 ​

我的使用模式是：在看到模型跑了很久、上下文很大时，按 Ctrl+C 中断了 agent 的运行。中断后继续输入新任务。

查看会话文件，最后一条 assistant 消息：

json
{
  "role": "assistant",
  "stopReason": "aborted",
  "usage": { "input": 0, "output": 0, "cacheRead": 0, "cacheWrite": 0, "totalTokens": 0 }
}

stopReason = "aborted" 表示被用户终止了，而且 usage 全为零。

再看 _checkCompaction 中是怎么处理带 usage 的正常消息和被终止的消息的：

javascript
// 正常消息（stopReason 不是 error）：
let contextTokens = calculateContextTokens(assistantMessage.usage);
// totalTokens = 0 → contextTokens = 0

if (shouldCompact(contextTokens, contextWindow, settings)) { ... }
// shouldCompact(0, 300000, { reserveTokens: 16384 })
// → 0 > 283616 → false → 不触发！

问题就在这里！ 如果用户中断了 agent，最后一条 assistant 消息的 totalTokens = 0，导致压缩检查永远通过不了。

但一轮（turn）中有很多条 assistant 消息（每个 tool call 来回产生一条），这里只检查最后一条。如果最后一条是 aborted，前面的高 token 消息就全被跳过了。

第五步：继续输入新任务后呢？ ​

继续看 prompt() 方法：

javascript
// Check if we need to compact before sending (catches aborted responses)
const lastAssistant = this._findLastAssistantMessage();
if (lastAssistant && (await this._checkCompaction(lastAssistant, false))) {
    // 触发压缩
}

这里的 _findLastAssistantMessage() 会去 agent 的 state 中找最后一条 assistant 消息。如果用户中断后继续输入，agent.state.messages 中最后一条仍然是那条 aborted 消息，usage.totalTokens = 0，压缩仍然不会触发。

但如果用户中断前，模型已经完成了一条正常的 assistant 响应（比如 toolUse），然后用户才中断的是后续的 agent 继续执行（continue()），那 _lastAssistantMessage 或者 agent.state.messages 中最后一条消息是什么，就取决于具体的执行时序。

第六步：后知后觉 ​

回头再看 Pi 的自动压缩设计文档，下面这段注释早就说明了问题：

markdown
After compaction, the last assistant usage reflects pre-compaction context size.
We can only trust usage from an assistant that responded after the latest compaction.
If no such assistant exists, context token count is unknown until the next LLM response.

这说的是压缩后的场景，但它也暗示了一个核心设计：Pi 通过 assistant 消息的 usage 来判断上下文大小，而一个被中断的 assistant 消息没有有效 usage，自然无法触发压缩。

最后结论 ​

根本原因并不复杂：

  1. 用户中断 agent → 最后一条 assistant 消息被标记为 aborted，usage 全为零
  2. 自动压缩检查依赖于 assistant 消息的 usage.totalTokens
  3. totalTokens = 0 永远满足不了 shouldCompact() 的阈值条件
  4. 如果继续输入新消息，prompt() 会再次检查——但如果最后一条 assistant 仍然是那条中断消息，压缩依然不会触发

最后一条 assistant 消息在 agent.state.messages 中的表现取决于中断的精确时刻：是在 tool call 执行中被中断，还是在模型的 assistant 响应中被中断，都会影响 state.messages 中保存的是什么。但无论如何，只要用户中断操作的发生时机使得最后一条有效 assistant 消息（带有高 token count 的）没有被设为 _lastAssistantMessage，压缩就不会被触发。

为什么没发现更长 ​

某种程度上这也是一件好事——说明 Pi 的中断处理很流畅，用户体验上没有卡顿感，所以一直没有意识到后台压缩没有正确触发。如果不是偶尔翻会话文件人工检查，可能还要过更久才会发现。

最终的修复方式和思考：

  1. 手工 /compact 是最直接的解决路径，永远可用
  2. 在中断恢复后，检查压缩条件时，应该追溯最后一条有有效 usage 的 assistant 消息，而不是只检查最后一条中断消息本身
  3. 配置层面可以调整 keepRecentTokens 让它更激进，但这不能解决中断场景下的根本问题

排查的价值不在于找到一个惊天大 Bug，而在于完整理解了一个看似"自动"的功能背后的工程权衡和边界条件。
